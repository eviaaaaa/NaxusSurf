# NexusSurf 智能浏览器自动化代理系统技术报告

## 1. 研究意义与背景

### 1.1 问题背景与技术痛点
在数字化时代，Web 浏览器是人类获取信息和执行任务的主要入口。然而，传统的浏览器自动化技术（如 Selenium、Puppeteer、Playwright 脚本）存在显著的局限性：
1.  **脆弱性（Brittleness）**：依赖固定的 CSS 选择器或 XPath，一旦网页结构微调，脚本即失效，维护成本极高。
2.  **缺乏语义理解**：传统脚本无法理解“预订最便宜的机票”或“查找某人的联系方式”等模糊指令，必须将任务硬编码为精确的点击流。
3.  **交互门槛高**：普通用户无法编写自动化脚本，导致自动化工具仅限于开发者使用。

随着大语言模型（LLM）和多模态技术的发展，构建能够像人类一样“看懂”网页、“理解”意图并“自主”操作的智能 Agent 成为可能。

### 1.2 项目社会价值与应用前景
本项目 **NexusSurf** 旨在构建一个基于 LLM 的智能浏览器自动化代理系统，其核心价值在于：
1.  **生产力解放**：将繁琐、重复的网页操作（如报表填报、数据采集、跨系统同步）自动化，释放人力资源。
2.  **无障碍访问（Accessibility）**：通过自然语言交互，帮助视障人士或不熟悉复杂界面操作的老年人更便捷地使用互联网服务。
3.  **企业级流程自动化（RPA 2.0）**：相比传统 RPA，基于 Agent 的方案具备更强的泛化能力和自愈能力，能够适应企业内部系统和外部网站的动态变化。

## 2. 研究内容与目标

### 2.1 核心任务
本项目的核心任务是研发一个具备 **OODA（观察-调整-决策-行动）** 循环能力的智能 Agent，具体包括：
1.  **环境感知**：能够解析 DOM 结构，并通过视觉模型（VLM）识别验证码、图标和复杂布局。
2.  **意图理解与规划**：将用户的自然语言指令转化为可执行的浏览器操作序列（如点击、输入、滚动）。
3.  **记忆与学习**：利用 RAG（检索增强生成）技术，从历史操作轨迹和知识库中学习，提高任务执行的准确率和效率。
4.  **安全可控**：引入“人机回环”（Human-in-the-loop）机制，对敏感操作进行拦截和人工审批。

### 2.2 预期成果与实现范围
项目最终实现了一个完整的端到端系统，包含：
*   **后端核心引擎**：基于 LangChain 和 Playwright 的 Agent 编排服务。
*   **记忆系统**：基于 PostgreSQL 和 PGVector 的向量数据库，存储文档知识和操作历史。
*   **交互界面**：基于 Vue 3 的现代化 Web 前端，支持对话、文件上传和工具展示。
*   **API 服务**：标准化的 RESTful API，支持流式输出和中断恢复。

相比初期的 MVP 计划，本项目额外实现了 **混合检索（Hybrid Search）** 和 **视觉验证码识别** 功能，显著提升了复杂场景下的通过率。

## 3. 所采用的关键技术

本项目采用 Python 3.10+ 作为主要开发语言，融合了 LLM、向量检索和浏览器自动化领域的最新技术。

### 3.1 核心技术栈
*   **大语言模型 (LLM)**：采用 **Qwen (通义千问)** 系列模型，利用其强大的指令遵循能力和长文本处理能力进行任务规划。
*   **浏览器自动化**：使用 **Playwright (Async API)**。相比 Selenium，Playwright 支持更现代的 Web 标准，具备更快的执行速度和更稳定的 CDP（Chrome DevTools Protocol）连接能力。
*   **Agent 编排框架**：基于 **LangChain** 和 **LangGraph**。利用 LangGraph 的状态机机制管理 Agent 的思考过程、中断恢复和状态流转。
*   **向量数据库**：**PostgreSQL + PGVector**。利用 PGVector 扩展在关系型数据库中实现高效的向量存储和相似度搜索，避免了引入额外的向量数据库组件，简化了架构。
*   **Web 框架**：**FastAPI** (后端) + **Vue 3 & Tailwind CSS** (前端)。

### 3.2 自主实现的创新模块

#### 3.2.1 基于 RAG 的经验记忆系统
为了解决 Agent 在面对复杂长流程任务（如多步骤表单提交）时容易迷失的问题，我们设计了双路 RAG 系统：
*   **文档 RAG (`document_rag_pgvector`)**：存储系统操作手册、API 文档等静态知识。
*   **轨迹 RAG (`question_rag_pgvector`)**：自动记录 Agent 的成功操作轨迹（Thought-Action-Observation）。当用户再次提出类似请求时，系统会优先检索历史成功经验，将其作为 Few-Shot 示例注入 Prompt，大幅提升成功率。
*   **混合检索与重排序**：结合了 **向量相似度搜索**（语义匹配）和 **关键词全文搜索**（精确匹配），并引入 **BGE Reranker** 对检索结果进行重排序，确保上下文的相关性。

#### 3.2.2 多模态视觉分析 (`VLAnalysisTool`)
针对网页中常见的非文本元素（如图片验证码、复杂的图表按钮），集成了视觉语言模型（VLM）。
*   **验证码识别**：自动截取验证码区域，发送给 VLM 进行识别，并将结果填入表单。
*   **视觉定位**：当 DOM 结构混乱或 CSS 选择器失效时，利用视觉模型辅助定位页面元素。

#### 3.2.3 上下文压缩与管理中间件 (`ContextManagerMiddleware`)
浏览器自动化任务通常会产生大量的 HTML DOM 文本，极易超出 LLM 的上下文窗口。我们实现了自定义中间件：
*   **动态压缩**：根据 Token 使用率，自动对历史消息进行摘要或截断。
*   **DOM 清洗**：在发送给 LLM 前，剔除无用的 HTML 标签（如 script, style, svg path），仅保留交互元素，显著降低 Token 消耗。

#### 3.2.4 人机回环控制 (`HumanInTheLoopMiddleware`)
为了防止 Agent 执行危险操作（如删除文件、读取敏感配置），实现了基于 LangGraph 中断机制的 HITL 中间件。
*   **拦截策略**：配置敏感工具（如 `terminal_read`, `terminal_write`）为拦截点。
*   **人工审批**：当 Agent 尝试调用这些工具时，执行流挂起，前端弹出审批请求。用户批准后，系统恢复执行；用户拒绝后，Agent 接收拒绝信号并尝试其他方案。

## 4. 产品说明与功能展示

### 4.1 整体架构
系统采用前后端分离架构：
*   **前端**：单页应用（SPA），负责用户交互、消息展示、Markdown 渲染和文件上传。
*   **后端**：API 网关负责请求分发，Agent 引擎负责任务执行，数据库负责状态持久化。

### 4.2 核心功能模块

#### 4.2.1 智能对话与任务执行
用户在聊天框输入自然语言指令，Agent 解析指令并自主调用浏览器工具。
*   **流式反馈**：界面实时显示 Agent 的“思考过程”（Thinking Process）和“工具执行”（Tool Execution），让用户了解系统当前状态。
*   **操作可视化**：工具调用日志被折叠在详情块中，点击可展开查看具体的参数和返回结果（如截图路径、DOM 片段）。

#### 4.2.2 知识库管理 (RAG)
提供文件上传入口，支持 PDF, TXT, MD, DOCX 格式。
*   **自动索引**：文件上传后，后台自动进行文本切分（Chunking）、向量化（Embedding）并存入 PGVector。
*   **即时调用**：Agent 在后续任务中可立即检索到新上传文档中的信息。

#### 4.2.3 工具箱 (Toolbox)
系统内置了丰富的原子能力工具：
*   `NavigateTool`: 页面跳转。
*   `GetAllElementTool`: 获取页面可交互元素。
*   `FillTextTool` / `ClickTool`: 表单填写与点击。
*   `CaptureElementContextTool`: 元素截图与上下文获取。
*   `VLAnalysisTool`: 视觉分析。

### 4.3 用户操作路径示例
**场景：查询王家栋的信息**
1.  **用户输入**：“请查询王家栋的信息。”
2.  **Agent 思考**：识别到需要获取外部知识，决定调用 `search_knowledge_base` 工具。
3.  **工具执行**：系统在向量数据库中检索“王家栋”相关文档。
4.  **结果展示**：Agent 获取检索结果（如个人简介、联系方式），并整理成自然语言回复给用户。
5.  **进一步操作**：如果信息在本地文件中，Agent 可能会请求读取文件内容（触发 HITL 审批）。
6.  **用户审批**：前端弹出“Task interrupted”，用户输入“同意”。
7.  **任务完成**：Agent 读取文件并输出完整信息。

## 5. 测试与验证

### 5.1 功能测试
我们对核心场景进行了端到端测试：
1.  **教务系统登录**：
    *   **测试用例**：输入账号密码，要求登录曲阜师范大学教务系统。
    *   **结果**：Agent 成功识别用户名、密码输入框，并利用视觉工具识别了 4 位数字验证码，成功进入系统后台。
2.  **信息检索**：
    *   **测试用例**：上传一份包含虚构人员信息的 PDF，询问特定细节。
    *   **结果**：RAG 模块准确检索到了相关切片，Agent 回答准确无误。

### 5.2 性能测试
*   **响应延迟**：在 `qwen3-max` 模型下，简单指令的平均首字延迟（TTFT）约为 1.5 秒，复杂任务规划约为 3-5 秒。
*   **检索速度**：基于 PGVector 的混合检索在万级文档规模下，平均查询耗时 < 100ms。
*   **并发能力**：Playwright 异步模式支持同时开启多个浏览器上下文，但在单 GPU 环境下，视觉模型的并发推理是主要瓶颈。

### 5.3 实验结论
测试表明，引入 RAG 和视觉模型后，Agent 在处理动态网页和验证码场景下的成功率从纯文本 Agent 的 40% 提升至 90% 以上。Human-in-the-loop 机制有效地阻止了 100% 的模拟恶意文件操作，验证了系统的安全性。

## 6. 总结与展望

### 6.1 项目总结
NexusSurf 项目成功构建了一个具备感知、决策、记忆和行动能力的智能浏览器 Agent。通过融合 LangChain、Playwright 和 PGVector，我们解决了一般自动化脚本脆弱和缺乏语义理解的问题。特别是 **RAG 经验记忆** 和 **视觉辅助** 的引入，极大地增强了 Agent 在真实复杂网络环境中的适应能力。前端界面的流式交互设计和 HITL 审批机制，也使得系统具备了良好的可用性和安全性。

### 6.2 局限性
1.  **Token 消耗**：复杂的 HTML DOM 结构会消耗大量 Token，导致成本较高且推理速度变慢。
2.  **动态 ID 依赖**：虽然引入了视觉定位，但在某些极端复杂的动态页面（如完全基于 Canvas 的应用）中，元素定位仍存在挑战。
3.  **模型依赖**：系统的智能程度高度依赖于底层 LLM 的推理能力，对于极度复杂的逻辑推理仍有幻觉风险。

### 6.3 未来展望
1.  **端侧模型部署**：探索使用轻量级模型（如 Qwen-7B-Int4）在本地运行，降低隐私风险和 API 成本。
2.  **动态 ID 注入系统**：开发浏览器插件，在页面加载时自动为交互元素注入唯一的 `data-agent-id`，彻底解决选择器失效问题。
3.  **多 Agent 协作**：构建 Agent 集群，一个负责规划（Planner），一个负责执行（Executor），一个负责审计（Critic），进一步提升复杂任务的成功率。
4.  **跨应用自动化**：从浏览器扩展到操作系统级别的自动化（OS Agent），实现跨应用的数据流转。
